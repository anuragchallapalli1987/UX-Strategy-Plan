03

Phase 1: Architectural Foundations (Weeks 1-3)

Close the "technical architecture knowledge" gap. You don't need to become an engineer — you need to think and communicate like an architect who happens to specialize in user experience.

Week 1

Platform & System Architecture for UX

Understand how modern platforms are built so you can architect UX that aligns with technical reality
▼

Learning Objectives

  • Understand micro-frontend architecture and its impact on UX consistency
  • Learn API-first design principles and how they shape interaction patterns
  • Map design system components to technical component architecture (Web Components, React, etc.)
  • Understand the role of a UX Architect vs. UX Designer vs. UI Engineer

Study Materials

  • Book: "Designing Web APIs" by Brenda Jin, Saurabh Sahni & Amir Shevat — Chapters 1-4 on API design patterns
  • Article: Martin Fowler — "Micro Frontends" (martinfowler.com) — essential reading
  • Article: Nathan Curtis — "Picking Parts, Products & People" (Medium) — system architecture for design
  • Video: "Design Systems as Code" — Clarity Conference talks (YouTube)
  • Hands-on: Diagram a micro-frontend architecture for Providence's product suite; map how your design system components would render across micro-frontends

Deliverable

Create a one-page architecture diagram showing how a UX architecture layer sits between business strategy and technical implementation — using Providence as the case study.

Week 2

Component Architecture & API Contracts

Think in terms of component APIs, contracts, and composability — the architect's language
▼

Learning Objectives

  • Define component API contracts (props, states, events, slots) — the bridge between design and engineering
  • Understand atomic design at the architectural level: atoms → molecules → organisms → templates → pages
  • Learn the DTCG (Design Token Community Group) token format specification
  • Study how design tokens flow from Figma → build pipeline → multi-platform rendering

Study Materials

  • Spec: DTCG Design Token Format specification (designtokens.org) — read the full spec
  • Book: "Atomic Design" by Brad Frost — re-read through an architect's lens (focus on Chapter 5: Maintaining Design Systems)
  • Article: "Design Tokens — Scaling Design with a Single Source of Truth" — CSS-Tricks
  • Article: "Component API Design" — Robin Rendle, designsystems.com
  • Hands-on: Define the component API contract for 5 key components from your Providence design system (Button, Modal, Form Field, Card, Navigation) — include props, variants, states, accessibility requirements, and responsive behavior

Deliverable

A component API specification document for 5 components with props tables, state machines, accessibility contracts, and platform-specific rendering notes.

Week 3

Performance Architecture & Core Web Vitals

Architects own performance as a design constraint, not an afterthought
▼

Learning Objectives

  • Understand Core Web Vitals (LCP, FID/INP, CLS) and how UX architecture decisions impact each
  • Learn performance budgets as design constraints — how to bake performance into design specs
  • Study image/asset architecture: lazy loading, responsive images, SVG systems, icon sprite strategies
  • Understand how design system bundle size impacts application performance

Study Materials

  • Course: web.dev — "Learn Core Web Vitals" (free, Google) — complete the full course
  • Article: "Designing for Performance" by Lara Hogan — key chapters on performance budgets
  • Article: "The Cost of JavaScript" by Addy Osmani (V8 blog)
  • Tool: Run Lighthouse audits on 3 healthcare platforms; document UX architecture decisions that would improve scores
  • Hands-on: Create a "Performance Architecture Spec" template that can be included in every design spec

Deliverable

A Performance Architecture Checklist that a UX Architect would include with every major design deliverable — covering loading strategy, asset optimization, animation performance, and CWV targets.

Phase 2: Enterprise Systems Mastery (Weeks 4-6)

Elevate your existing design systems experience to architectural scale — multi-brand, multi-platform, federated governance.

Week 4

Design System Architecture at Enterprise Scale

Multi-brand, multi-platform, federated contribution models
▼

Learning Objectives

  • Study federated design system models (centralized vs. federated vs. hybrid governance)
  • Learn multi-brand theming architecture (how one system serves 5+ product brands)
  • Understand design system maturity models and how to assess organizational readiness
  • Study real-world enterprise systems: Salesforce Lightning, IBM Carbon, Atlassian Design System, Microsoft Fluent

Study Materials

  • Book: "Design Systems" by Alla Kholmatova (Smashing Magazine) — the architecture chapters
  • Case Study: Salesforce Lightning Design System documentation — study governance model
  • Case Study: IBM Carbon — study the contribution model and multi-framework support
  • Article: "The Sliding Scale of Design Systems" — Nathan Curtis
  • Article: "Design System Governance" — InVision blog
  • Hands-on: Create a maturity assessment of your Providence design system using a standard maturity model
Week 5

Cross-Platform UX Architecture

Web, mobile, voice, spatial — one coherent experience architecture
▼

Learning Objectives

  • Architect platform-agnostic experience patterns that adapt across web, iOS, Android, and emerging platforms
  • Study adaptive vs. responsive architecture — when each is appropriate
  • Understand spatial computing UX architecture (visionOS, XR interfaces)
  • Learn voice UX architecture patterns and multimodal interaction design

Study Materials

  • Guidelines: Apple Human Interface Guidelines — Spatial Design section (developer.apple.com)
  • Guidelines: Google Material Design 3 — Adaptive Layouts documentation
  • Book: "Designing Connected Products" by Claire Rowland — cross-platform ecosystems
  • Course: NN/g "Omnichannel UX" course — understand multi-touchpoint architecture
  • Hands-on: Architect a cross-platform experience map for a healthcare patient portal (web → mobile → wearable → voice)
Week 6

Accessibility Architecture & Automation

Move from compliance to architectural embedding — automated, systematic, proactive
▼

Learning Objectives

  • Design accessibility as an architectural pattern, not a testing afterthought
  • Build automated accessibility pipelines (axe-core in CI/CD, Storybook a11y addon, Playwright a11y tests)
  • Study WCAG 3.0 draft and the shift from conformance levels to scoring
  • Architect accessible design tokens (contrast ratios, focus styles, reduced-motion) as system defaults

Study Materials

  • Spec: WCAG 3.0 Working Draft (W3C) — understand the new scoring model
  • Tool: Deque University — "Accessibility Architecture" course (free tier)
  • Article: "Accessibility in Design Systems" — A List Apart
  • Hands-on: Design an "Accessibility Architecture Spec" for your design system — covering token-level a11y, component-level ARIA contracts, and pipeline-level automated testing
05

Phase 3: Strategic Leadership (Weeks 7-9)

The differentiator between a senior practitioner and an architect is the ability to set multi-year vision, align with business strategy, and communicate at the executive level.

Week 7

UX Architecture Vision & Strategy

Articulate a 3-5 year UX architecture roadmap aligned to business objectives
▼

Learning Objectives

  • Create a UX Architecture Strategy document — vision, principles, capability roadmap, investment thesis
  • Learn Wardley Mapping for UX — map UX capabilities on the evolution axis to identify investment priorities
  • Study how to align UX architecture investments with business KPIs (revenue, retention, efficiency, compliance)
  • Practice writing architecture decision records (ADRs) for UX

Study Materials

  • Book: "Team Topologies" by Matthew Skelton & Manuel Pais — Chapters on team architecture
  • Framework: Wardley Mapping — "Learn Wardley Mapping" (learnwardleymapping.com) — first 5 chapters
  • Article: "Architecture Decision Records" — Michael Nygard (cognitect.com)
  • Hands-on: Write a 3-year UX Architecture Vision for Providence — covering design system evolution, AI integration roadmap, platform expansion, and accessibility maturity
Week 8

AI-Native UX Architecture

Architect interfaces that are AI-native, not AI-bolted-on
▼

Learning Objectives

  • Understand AI-native UX patterns: adaptive interfaces, predictive navigation, generative UI, conversational UX
  • Architect for uncertainty — how to design systems where AI outputs are non-deterministic
  • Study human-AI interaction patterns: confidence indicators, explanation layers, override mechanisms, feedback loops
  • Learn the compound intelligence model — AI augments every PDLC phase

Study Materials

  • Guidelines: Google PAIR — "People + AI Guidebook" (pair.withgoogle.com) — complete the full guidebook
  • Guidelines: Apple — "Human Interface Guidelines: Machine Learning" section
  • Article: "Designing Human-AI Experiences" — Microsoft Research
  • Book: "Designing Human-Centric AI Experiences" by Akshay Sharma (O'Reilly)
  • Hands-on: Architect an AI-enhanced patient triage flow showing: confidence indicators, human override points, explanation layers, and graceful degradation when AI is uncertain
Week 9

Executive Communication & Org Design

Sell architecture to executives; design the team that delivers it
▼

Learning Objectives

  • Frame UX architecture investments in business language (ROI, risk reduction, velocity improvement, compliance)
  • Design organizational structures for UX architecture teams (embedded vs. platform vs. center of excellence)
  • Create capability models and hiring frameworks for architecture talent
  • Practice the "architecture pitch" — a 10-minute executive presentation of a UX architecture transformation

Study Materials

  • Book: "Org Design for Design Orgs" by Peter Merholz & Kristin Skinner — the definitive guide
  • Article: "The UX of UX Architecture" — Peter Morville (intertwingled.com)
  • Practice: Write a 1-page "Architecture Investment Proposal" for an executive audience — covering problem, proposed architecture, expected ROI, timeline, and risks
  • Practice: Record yourself delivering the 10-minute architecture pitch; review and iterate 3 times

Phase 4: Interview Mastery (Weeks 10-12)

Apply everything you've learned. Mock interviews, case study practice, whiteboard sessions, and portfolio narrative polish.

Week 10-12

Mock Interviews, Case Studies & Portfolio

3 weeks of intensive interview simulation
▼

Week 10: Case Study Deep Practice

  • Complete 3 full case study exercises (45-60 min each) using the framework in Section 08
  • Record yourself presenting each case; review for clarity, executive presence, and architecture depth
  • Get peer feedback from at least 2 senior designers or architects

Week 11: Mock Interview Rounds

  • Schedule 3 mock interviews — one behavioral, one technical, one whiteboard
  • Use the 80+ question bank (Section 07) to prepare; rehearse answers for the top 20
  • Practice the STAR-A method: Situation, Task, Action, Result, Architecture Insight

Week 12: Portfolio & Narrative Polish

  • Finalize portfolio presentation (3-4 case studies, architecture-framed)
  • Prepare "Tell me about yourself" as a 90-second architecture career narrative
  • Prepare questions to ask interviewers (shows architectural thinking)
  • Final dress rehearsal — full interview simulation with a mentor

Case Study Framework

Principal UX Architect interviews often include a 45-60 minute case study presentation. Use this framework — and specifically prepare your Providence and WinWire stories in this structure.

The ARCH Framework (Architecture Case Presentation)

🔍
A — Assess
2-3 minutes
What was the architectural landscape? Audit the existing state: tech stack, design maturity, team structure, user scale, compliance requirements. Show you diagnose before you prescribe.
📐
R — Recommend
5-7 minutes
What architecture did you propose? Design system strategy, component model, token architecture, governance framework, platform considerations. Show the WHY behind every decision.
🔧
C — Construct
5-7 minutes
How did you build it? Phased rollout, team structure, adoption strategy, stakeholder management, technical implementation approach. Show pragmatic execution.
📊
H — Harvest
3-5 minutes
What were the measurable outcomes? Metrics (conversion, velocity, adoption, satisfaction), business impact, lessons learned, what you'd do differently. Show you measure architecture success.

Portfolio Positioning Guide

Your portfolio needs to shift from "I designed this" to "I architected this system." Here's how to reframe each case.

Portfolio Reframing: Designer → Architect

Designer Portfolio SaysArchitect Portfolio Says
"I designed the onboarding flow""I architected the onboarding experience pattern that scales across 10 products and adapts to 3 user personas, with progressive disclosure rules codified in the design system"
"I built the design system""I defined the token architecture, component API contracts, governance model, and contribution workflow for a federated design system serving 15 product teams"
"I made it accessible""I embedded accessibility as an architectural layer: ARIA contracts at the component level, contrast tokens at the system level, and automated a11y testing in the CI/CD pipeline"
"I improved conversion by 25%""I identified the architectural root cause of conversion drop-off (inconsistent interaction patterns across 3 product modules), unified the interaction architecture, and measured a 25% improvement"
"I integrated AI tools into design""I established the AI-augmented design architecture layer: tool selection framework, workflow integration patterns, quality gates for AI-generated outputs, and team adoption strategy"

Your 90-Second Architecture Career Narrative

Practice This Daily
"I'm an enterprise UX architect with 16 years of experience transforming how organizations approach design at scale. I started in front-end development, which gives me the technical fluency to architect systems that engineering teams can actually build. Over the past decade, I've progressively moved from designing interfaces to architecting enterprise-scale design systems — most recently at Providence, where I architect the UX layer across 10+ healthcare products serving Medicare/Medicaid populations. My work combines three things that are hard to find together: deep accessibility expertise baked into architectural patterns, AI-integrated design workflows, and enterprise design governance across 15-20 person cross-functional teams. I'm looking for a Principal UX Architect role where I can define the architectural vision for a product platform — not just the pixels, but the systems, patterns, governance, and standards that let an entire organization deliver consistent, accessible, high-quality experiences at scale."

30 min

30 min

30 min
